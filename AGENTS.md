# Repository Guidelines

## プロジェクト構成とモジュール構造
ルート直下の `parser.go` が公開 API をまとめ、同階層の `parser_test.go` でパーサ挙動を検証します。`models/` には各言語向けの学習済みモデル (`ja.go`, `zh-hans.go` など) と `model.go` があり、シリアライズ済みの重みと読み出しロジックを保持します。モデル更新を自動化するスクリプトは `gen/generate_models.go` に配置されており、生成物は `models/` に書き戻されます。

## ビルド・テスト・開発コマンド
開発中の最小検証には `go test ./...` を使用してください。動作を詳細確認したい場合は `go test -v ./...` を推奨します。ライブラリとしてのエクスポート確認には `go build ./...` を実行し、モジュール境界での依存解決を検証します。モデルを再生成する際は `go run gen/generate_models.go` を実行し、コミット前に生成物差分を確認してください。

## コーディングスタイルと命名規約
Go の標準スタイルに従い、インデントはタブ、行末に不要な空白を残さないでください。保存前に `gofmt` または `go fmt ./...` を必ず適用し、パッケージインポートは `goimports` で整形することを推奨します。公開関数・型にはパッケージコメント規約を満たす英語 Doc コメントを添え、テスト補助関数など内部利用に限る識別子は先頭小文字で定義してください。モデル名やロケールコードは既存ファイルと同じケバブケース (`zh-hans`) を踏襲します。

## テスト方針
標準ライブラリの `testing` パッケージを活用し、テーブルドリブンテストで入力と期待セグメントを明示してください。テスト関数名は `Test` で始め、対象領域を表すサフィックス (例: `TestParser_Parse`) を付けます。カバレッジ目標は 90% 以上を目安とし、`go test -cover ./...` で測定して PR 説明に記載してください。学習モデル更新時は既存サンプルを網羅するリグレッションテストを追加し、言語別差分を `models/model_test.go` に集約します。

## コミットおよびプルリクエスト方針
`main` ブランチは常にリリース可能な状態を保つため、機能追加や修正はトピックブランチで行ってください。コミットメッセージは Conventional Commits (`feat:`, `fix:`, `docs:` など) を採用し、1 行目で変更概要を簡潔に示したうえで必要に応じて本文に背景と影響範囲を記載します。関連 Issue がある場合は `Refs #123` や `Fixes #123` 形式で紐付け、複数コミットの PR では履歴を読みやすく保つため適宜 `rebase -i` で整えます。プルリクエスト本文には変更点の箇条書き、実行したテストコマンド、モデルファイルに差分がある場合のサマリ (言語・サイズ・生成手順) を記載し、スクリーンショットやサンプル出力が再現確認を助ける場合は添付してください。

## モデル更新と配布に関する留意事項
新しいモデルを取り込む際は配布ライセンスとソースを記録し、`models/` に含まれる生成ファイルのみを更新対象とします。生成スクリプトに依存する外部データはリポジトリ外置きを基本とし、再現手順を `README.md` または該当 PR に記載することで長期的な保守性を確保してください。
